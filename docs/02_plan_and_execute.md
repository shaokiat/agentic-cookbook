# Plan-and-Execute Agent Pattern

The **Plan-and-Execute** pattern is a two-phase architectural approach that separates high-level strategy from low-level execution. This is particularly effective for complex goals that require multiple distinct steps.

## Overview

Unlike a pure **ReAct** agent that decides its next move on the fly, Plan-and-Execute uses a "Planner" to decompose a goal into a concrete list of tasks, which are then carried out by an "Executor."

1.  **Phase 1: Planning**: A model is tasked with generating a structured plan without executing any tools.
2.  **Phase 2: Execution**: A separate agent (or the same model with a different prompt) follows the plan step-by-step, using the standard thinking-acting loop.

---

## Architecture

This pattern is implemented in [examples/02_plan_and_execute.py](../examples/02_plan_and_execute.py).

### The Planner
The planner is a simple model call with a system prompt that enforces strategic thinking. It produces a numbered list of steps which are passed to the executor as its primary instruction set.

### The Executor
The executor is an instance of the [Agent](../core/agent.py) class. It is initialized with the generated plan as part of its system prompt, focusing it entirely on carrying out the planned steps using its available tools.

---

## The Agent Call Loop

The **Execution Phase** relies on the core recursive loop defined in `Agent.run`. This is where the actual "agentic" behavior occurs.

### The Think-Act-Observe Cycle
The loop (often called the "Heartbeat" of the agent) follows a sequential cycle:

1.  **Think**: The model received the conversation history + tool schemas and decides the next action (text response or tool calls).
2.  **Act**: If the model issues tool calls, the [ToolRegistry](../core/registry.py) executes the corresponding Python functions.
3.  **Observe**: Content generated by the model and tool outputs are fed back into the [Memory](../core/memory.py).
4.  **Repeat**: The loop cycles back to **Think**, now with the benefit of the new observation.

### Code Snippet: The Loop Logic
```python
while steps < self.max_steps:
    # 1. Think (Model Generation)
    response = self.model.generate(
        messages=self.memory.get_messages(),
        tools=self.registry.get_schemas()
    )

    # 2. Observe (Handle Content)
    if response.content:
        self.memory.add_message("assistant", response.content)

    # 3. Act (Tool Execution)
    if response.tool_calls:
        # Loop continues if tools were called
        for tool_call in response.tool_calls:
            result = self.registry.call_tool(...)
            self.memory.add_message("tool", result, ...)
        continue 
    else:
        # Loop breaks if no tool calls (Final Answer)
        break 
```

## Deep Dive: Instrumentation & Analysis

To understand the internal mechanics, we enabled structured logging by passing a `log_path` to the agent initialization. 

### Step-by-Step Breakdown (Run Analysis)

Following the run log [`agent_run.log`](../agent_run.log) generated during a project analysis task:

#### 1. Exploration (Step 1)
- **Rationale**: The agent identifies the need to verify the local project structure.
- **Action**: Calls `list_files(directory="./")`.

#### 2. Deep Dive (Step 2)
- **Rationale**: Having confirmed `core/` exists, the agent "enters" the directory to list its source files.
- **Action**: Calls `list_files(directory="core/")`.

#### 3. Parallel Execution (Step 3)
- **Rationale**: The agent identifies 4 critical source files. Instead of 4 separate steps, it issues **4 parallel tool calls** in a single response to read them all at once.
- **Action**: `read_file_content` for `memory.py`, `registry.py`, `model.py`, and `agent.py`.

> [!NOTE]
> **Logical vs. Physical Parallelism**
> While the model identifies these actions "in parallel" (logical batching), the current `Agent` implementation executes them **sequentially** in a Python loop. The primary optimization here is reducing the number of LLM round-trips, not concurrent execution of the tool code itself.

- **Note**: This demonstrates how modern LLMs optimize token usage and latency by batching independent actions into a single turn.

#### 4. Final Synthesis (Step 5)
- **Rationale**: The agent has gathered all information from the source code and READMEs. It sees that its current internal state contains the complete answer.
- **Status**: Generates the summary and terminates the loop because `tool_calls` is empty.

### Quantification: How Many Times & Why?

| Metric | Value | Rationale |
| :--- | :--- | :--- |
| **Total LLM Calls (Steps)** | **5** | Number of times the model "thought" after receiving new observations. |
| **Total Tool Calls** | **8** | Higher than the step count because the agent used parallel batching. |
| **Termination Trigger** | `break` | Triggered automatically when the model provides a text-only final answer. |
| **Safety Limit** | **10** | A default "max_steps" to prevent infinite loops in case of model hallucination. |

## How to Reproduce
To generate these insights yourself, initialize the agent with the `log_path` parameter:

```python
agent = Agent(
    model=model,
    memory=Memory(),
    registry=registry,
    log_path="agent_run.log"
)
```

Running the script will now produce a JSONL file capturing the granular details of every LLM interaction and tool execution.
